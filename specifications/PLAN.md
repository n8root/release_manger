# План: Система управления релизами (Release Manager)

**Масштаб**: self-hosted решение для одной компании, ориентировано на максимум несколько тысяч пользователей.

---

## 0. Технологический стек и развёртывание

### Стек
| Компонент | Технология |
|-----------|------------|
| Backend | Laravel (PHP) |
| Frontend | Vue.js 3, Vite |
| База данных | PostgreSQL |
| Кэш / очереди / сессии | Redis |
| Развёртывание | Docker, docker-compose (self-hosted) |

### Требования к инфраструктуре
- Все сервисы запускаются в Docker; образы по возможности минимальные (Alpine).
- Приложение рассчитано на нагрузку в рамках одной компании (порядка тысяч пользователей), без требования горизонтального масштабирования на десятки инстансов.

### Состав Docker
- **app** — Laravel (PHP 8.5-FPM Alpine), сборка фронтенда через Vite на этапе сборки образа; Node в образ не входит после сборки.
- **nginx** — `nginx:alpine`, раздача `public/` и проксирование PHP на app.
- **pgsql** — `postgres:16-alpine`.
- **redis** — `redis:7-alpine` (сессии, кэш, очереди).
- **queue** — воркер очередей Laravel (тот же образ, что app); опционально для фоновых задач и уведомлений.
- Конфигурация: `docker-compose.yml`, `Dockerfile`, `.docker/nginx/default.conf`, `.dockerignore`.

## 1. Ролевая модель доступа и авторизация (RBAC)

### Роли в системе
| Роль | Описание и права |
|------|------------------|
| **Администратор** | Полный доступ к платформе, управление пользователями, настройками, интеграциями |
| **Владелец сервиса** | Деплой и откат релизов своего сервиса; апрув пунктов в чеклисте (например, план отката); ответственный за релиз |
| **Разработчик** | Создание/редактирование релизов, добавление MR, просмотр; без деплоя и отката |
| **Тестировщик** | Просмотр релизов, проставление проверок (в т.ч. ручных), участие в чеклистах |
| **Инцидент-менеджер** | Управление инцидентами по релизам: создание, закрытие, редактирование |

### Требования к RBAC
- Удобная и понятная модель: роли назначаются пользователям (глобально и/или в контексте команды/сервиса).
- Разграничение: только ответственный за релиз (назначенный владелец) может деплоить и откатывать конкретный релиз.
- Апрувы в чеклистах — согласно ролям (например, план отката апрувит владелец сервиса).

## 2. Релизы (Releases)

### Реестр релизов
- **Список релизов**: таблица/карточки с фильтрацией по статусу, сервису, команде, дате, ответственному.
- **Действия**: добавление, удаление, редактирование релиза.
- **Движение по статусам**: Черновик → Готов к релизу → Задеплоен → Откачен (с возможностью возврата в Черновик).
- **Оценка факторов риска**: отображается в UI; часть факторов вычисляется автоматически по данным из MR (в т.ч. через MCP/ИИ); часть — ручная; факторы, проставленные через MCP, можно удалять.
- **Лог пайплайнов**: на странице списка/детальной странице релиза отображается лог пайплайнов (сборка, деплой и т.д.).

### Детальная страница релиза
- Редактирование основных полей релиза.
- **Список MR**: добавление MR к релизу; через интеграцию (GitLab/GitHub и т.д.) MR собираются в релизную ветку.
- **Сборка в релизную ветку**: выполняется при переходе в статус «Готов к релизу»; при возврате в «Черновик» релизная ветка удаляется.
- **Просмотр diff**: возможность просмотра совокупного diff по всем MR релиза.
- **Инциденты**: список инцидентов, связанных с релизом.
- **Правила выкатки**: настраиваемые правила (катить после релиза X, катить после даты Y и др.); зависимости между релизами и правилами должны быть наглядно визуализированы (граф/дерево зависимостей).
- **Чеклист проверок**: список проверок, обязательных для перехода на следующий статус:
  - Часть проверок проставляется автоматически (например, «тесты запущены»).
  - Часть — вручную уполномоченным лицом (например, «валидный план отката» — владелец сервиса).
  - Релиз не переходит на следующий статус, пока не выполнены все необходимые проверки; при попытке перехода — понятное уведомление о незаполненных проверках.
- **Ответственный за релиз**: назначение ответственного (владелец сервиса/делегат); только он может деплоить и откатывать этот релиз.
- **Уведомления по релизу**: привязка правил нотификации к релизу; пресеты берутся из настроек команды (каналы + события), здесь задаются конкретные правила для данного релиза.
- **История изменений релиза**: полный журнал изменений по релизу — кто и когда менял поля (название, описание, ответственный, MR, чеклист, правила выкатки), смена статусов, деплой/откат. Отображается на детальной странице в виде ленты (дата, пользователь, действие, старое/новое значение). Используется для аудита и разбора инцидентов.

### Статусы релиза
- **Черновик** — редактирование, добавление MR; при возврате из «Готов к релизу» — удаление релизной ветки.
- **Готов к релизу** — релизная ветка собрана, выполнены проверки, можно деплоить.
- **Задеплоен** — релиз выкатан.
- **Откачен** — релиз откачен.

### Механика деплоя и отката
- **Триггер деплоя**: действие «Деплой» доступно только ответственному за релиз; при нажатии система вызывает внешний контур (API пайплайна CI/CD, webhook в систему деплоя — GitLab CI, GitHub Actions, ArgoCD и т.д.).
- **Ожидание результата**: после запуска деплоя отображается статус (в процессе / успех / ошибка); при необходимости — поллинг статуса пайплайна или приём webhook с результатом.
- **Откат**: аналогично — кнопка «Откат» только у ответственного; вызов API/пайплайна отката; результат отображается в логе и в истории изменений релиза.

### Версионирование релизов
- **Версия релиза**: источник версии — ручной ввод, автоинкремент по сервису или привязка к тегу в Git (semver, календарная версия и т.д.) — настраивается на уровне сервиса или платформы.
- **Связь с артефактами**: при необходимости — связь релиза с Git tag и/или артефактом сборки (образ, пакет) для трассируемости.

### Окружения (staging / production)
- Сервис может иметь несколько **окружений** (например, staging, production); деплой и откат выполняются в контексте выбранного окружения.
- Правила выкатки и чеклисты могут различаться по окружениям (например, для production — дополнительные проверки).
- В календаре и реестре релизов при необходимости отображается окружение.

### Шаблоны релизов и чеклистов
- **Шаблоны релиза**: пресеты с дефолтными правилами выкатки, наборами проверок и пресетами уведомлений; при создании релиза можно выбрать шаблон (по типу сервиса или команде).
- **Шаблоны чеклиста**: заготовки чеклиста по типу сервиса (бэкенд, фронт, критичный сервис и т.д.); привязка к команде/сервису в настройках.

### Постдеплой-проверки
- Опциональные проверки **после** деплоя: ручные (например, «Проверил в prod») или автоматические (health-check по интеграции).
- Релиз может переходить в финальный статус «Задеплоен» только после прохождения постдеплой-проверок (если они включены в настройках сервиса/чеклиста).

### Конфликты и ограничения выкатки
- **Конфликты**: при планировании и деплое проверяется, что на один и тот же день/окружение не запланировано несколько релизов одного сервиса без явного разрешения; при нарушении — предупреждение или блокировка.
- **Зависимости**: блокировки вида «сервис X не катить, пока не задеплоен релиз Y» проверяются перед разрешением деплоя.
- **Лимиты**: при необходимости — ограничения (не более N релизов в день по сервису/платформе); настраиваются в правилах выкатки или в настройках платформы.

## 3. Команды (Teams)

- **CRUD**: добавление, редактирование, удаление команд.
- **Каналы уведомлений**: настройка каналов (Slack, Telegram, и т.д.) для команды.
- **Пресеты уведомлений**: дефолтные пресеты событий и каналов для нотификаций команды.
- **Привязка сервисов**: каждый сервис привязан к одной команде.

---

## 4. Сервисы (Services)

- **CRUD**: добавление, удаление, редактирование сервисов.
- Сервис — единица деплоя (приложение/микросервис); привязка к команде обязательна.
- **Окружения**: у сервиса могут быть окружения (staging, production и т.д.); правила выкатки и чеклисты настраиваются при необходимости по окружению.

---

## 5. Пользователи (Users)

- **Реестр пользователей**: список пользователей системы.
- **CRUD**: добавление, удаление, редактирование пользователей.
- Связь пользователей с ролями и (опционально) с командами/сервисами для контекстного доступа.

---

## 6. Релизный календарь (Release Calendar)

- **Календарь**: вид «день/неделя/месяц» с отображением запланированных релизов по дням.
- **Правила выкатки по датам**: настройка правил — в какой день какой сервис можно/нельзя катить (например, «сервис X только по вторникам»).
- В календаре отображаются все запланированные на выбранный день релизы и учтённые ограничения по сервисам.

---

## 7. Реестр инцидентов (Incidents)

- **Список инцидентов**: реестр с фильтрацией (по релизу, сервису, статусу, дате).
- **CRUD**: добавление, удаление, редактирование инцидентов.
- Связь инцидента с релизом (и опционально с сервисом); на детальной странице релиза отображается список связанных инцидентов.
- События «открытие/закрытие инцидента» участвуют в событийной модели для уведомлений и аудита.
- **История изменений инцидента**: журнал изменений — создание, смена статуса, редактирование описания/титула/приоритета, привязка к релизу, комментарии. Отображается на странице инцидента; используется для аудита и постмортемов.

---

## 8. Настройки платформы (Settings)

- **Интеграции**: управление интеграциями (GitLab/GitHub, трекеры, MCP и т.д.) — добавление, удаление, редактирование, включение/выключение.
- **Боты для уведомлений**: настройка ботов для мессенджеров (Slack, Telegram, и др.).
- **Трекеры**: интеграция с Яндекс.Трекером и другими трекерами (задачи, тикеты).
- **Факторы риска**: справочник факторов риска — добавление, удаление, редактирование; часть факторов используется для автоматической оценки (в т.ч. через MCP/ИИ).
- **Событийная модель**: фиксированный набор событий (смена статуса релиза, редактирование релиза, открытие/закрытие инцидента и т.д.); на основе событий строятся правила уведомлений и (при необходимости) аудит.

---

## 9. События и уведомления

- **События**: смена статуса релиза, редактирование релиза, открытие/закрытие инцидента, деплой, откат и др.
- **Пресеты в команде**: какие каналы и по каким событиям уведомлять по умолчанию.
- **Правила по релизу**: для конкретного релиза можно переопределить/дополнить правила нотификации (какие события в какие каналы).
- Удобная настройка: выбор каналов и событий без дублирования логики.

---

## 10. Интеграции и MCP

- **Сбор MR в релизную ветку**: интеграция с Git (GitLab/GitHub) — при переходе в «Готов к релизу» создаётся релизная ветка и в неё мержатся выбранные MR; при возврате в «Черновик» ветка удаляется.
- **Факторы риска через MCP**: часть факторов риска анализируется ИИ через MCP-серверы; результаты отображаются в UI; пользователь может удалить автоматически проставленный фактор (с пометкой источника).
- **Пайплайны**: отображение логов пайплайнов (CI/CD) в контексте релиза (по интеграции с GitLab CI/GitHub Actions и т.д.).

### Внешний API
- **REST API** для интеграций: запуск деплоя/отката, получение статусов релизов, создание/обновление релизов из внешних систем (CI, скрипты).
- Версионирование API и документация (OpenAPI/Swagger).
- Доступ по токенам/ролям; только уполномоченные роли могут вызывать критические операции.

## 11. Дополнительные требования (уточнения и расширения)

### Безопасность и аудит
- Логирование критичных действий: смена статуса, деплой, откат, изменение прав, изменение правил выкатки.
- Сессии и авторизация: поддержка SSO/OAuth (по возможности) и хранение сессий с возможностью инвалидации.

### Валидации и ограничения
- Запрет перехода в «Готов к релизу» при невыполненных проверках чеклиста с явным сообщением.
- Запрет деплоя/отката не ответственным за релиз.
- Проверка правил выкатки (календарь, зависимости от других релизов) перед разрешением деплоя.

### UX
- Единый стиль уведомлений об ошибках и предупреждениях (тосты/модалки).
- Визуализация зависимостей релизов (граф/дерево) на детальной странице и в календаре.
- Адаптивный интерфейс для работы с планшетов и больших экранов.
- **Локализация**: язык интерфейса (русский/английский) — по умолчанию или по настройке пользователя/команды; при необходимости — локализация уведомлений и отчётов.

### Масштабируемость
- Архитектура, допускающая добавление новых интеграций (трекеры, мессенджеры, MCP) без переписывания ядра.
- Конфигурируемые факторы риска и проверки (справочники в настройках), а не зашитые в код.

### Отчёты и метрики
- **Метрики**: частота релизов (по сервису/команде/периоду), доля откатов, время от «Готов к релизу» до «Задеплоен», связь инцидентов с релизами (количество за период).
- **Отчёты и дашборды**: экраны для владельцев сервисов и менеджмента (сводки, тренды); при необходимости — экспорт в CSV/Excel.

### Импорт и миграция
- **Импорт данных**: при внедрении — возможность загрузить существующие релизы/инциденты из текущих инструментов (таблицы, Jira и т.д.) для сохранения истории.
- **Миграция пользователей и прав**: перенос пользователей, ролей и привязки к командам/сервисам из LDAP/SSO или предыдущей системы.

### Многошаговые апрувы (опционально)
- Помимо чеклиста — возможность настроить цепочки согласований (например, техлид → владелец продукта) для перехода в «Готов к релизу» или деплоя; при отсутствии требования достаточно ролевых проверок в чеклисте.
